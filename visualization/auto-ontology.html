What is a Fabric Ontology?

A Fabric Ontology is a semantic layer that sits on top of your Lakehouse tables. It translates raw data into named, typed business entities and relationships — the same concepts used in everyday business conversations.

  • Each Entity maps a Lakehouse table to a business concept (claims_fact → Claim)
  • Each Relationship wires entities together via foreign keys with a business name
  • The ontology is queried by the Data Agent to resolve natural language into GQL graph traversals
  • Runs are idempotent — re-running the notebook rebuilds from scratch safely

🤖
How the Data Agent Uses This

  1. 1
    User asks a natural-language question about claims, policies, or payments
  2. 2
    Agent resolves business terms against the ontology's named entities and relationships
  3. 3
    Agent generates a GQL graph query that traverses the entity relationships defined in the ontology
  4. 4
    Results are returned in business language — no GQL or schema knowledge required

The ontology is the bridge between human intent and table-level data.

Faster Multi-Hop Queries
GQL Traversal vs. SQL JOINs
  • SQL JOIN complexity grows exponentially with relationship depth — a 4-hop query requires 8+ nested JOINs that can take minutes to run
  • GQL graph traversal follows node-edge structures that are physically adjacent in storage, completing the same queries in seconds
  • GQL expresses business intent directly: MATCH (c:Customer)-[:holds]->(p:Policy)-[:covers]->(v:Vehicle) — far less likely to generate an incorrect query
  • The AI agent generates GQL that mirrors how a human thinks about the data

🛡️
↓ Hallucination
Grounded Answers Only
Structural Grounding Reduces AI Errors
  • Vector-only RAG fragments documents into chunks, severing entity relationships and causing the AI to infer connections that don't exist
  • The ontology graph preserves explicit relationships — the agent traverses defined paths rather than guessing
  • Ontology-based query checking catches semantically invalid traversals before execution and prompts the AI to repair them
  • When a question is unanswerable, the agent returns "I don't know" rather than fabricating a result
Bernardo et al. (arXiv 2025) · Tuteja et al. (OpenReview 2026) · Sequeda OBQC results

Single Source of Truth
Governed, Consistent Business Semantics
  • In enterprise data, the same concept ("customer", "active policy") often means different things in different systems — the AI hits them all and gives inconsistent answers
  • An ontology defines every business concept once, with a stable name, description, and data binding — used consistently across all agents, BI reports, and GQL queries
  • Properties like Adjusted Year-to-Date Revenue replace cryptic column names like cust_rev_ytd_adj
  • Data lineage is maintained for every entity and relationship — every answer is auditable back to its source tables
